Firebird .NET Provider

O poder do driver ADO.NET do firebird

·       Introdução

Pra quem ainda não conhece o Firebird.NET provider é um driver de acesso ao firebird para a plataforma Microsoft® .NET®, com esse driver é possível acessar o banco de forma bastante fácil, neste artigo irei ensinar passo a passo como instalá-lo integrando-o junto ao delphi for .NET., explicar suas principais funcionalidades e apresentar alguns exemplos práticos de como utilizá-lo desde a conexão ao banco até a manipulação de backups e usuários.

Em primeiro lugar você deve fazer o download do driver bem como do próprio firebird 1.5 caso já não o tenha. Você os encontra no site oficial: http://www.firebirdsql.org/ , até o final da edição deste artigo as últimas versões estáveis são a 1.7 do driver e a 1.5.3.x. do firebird.

 

·        Instalando o Firebird.NET Provider 1.7 e integrando seus componentes com o delphi 2005/2006

 

A instalação deste provider é bem simples, faça o download do arquivo de instalação e instale normalmente, nenhuma opção adicional será necessária, depois de instalado vamos integrar ele ao delphi para que possamos trabalhar em modo de design com seus componentes, abra o delphi e vá ao menu Component>Instaled .NET Components (lembre-se que se estiver usando o delphi 2006 a personalidade “Delphi for the Microsoft .NET Framework” deve ser carregada para que essa opção esteja disponível), na tela que se abre selecione a guia “.NET Components”.

Imagem 1

 

Logo abaixo de uma lista de assemblys .NET você encontra um campo “Add components” em Category digite “Firebird” sem aspas e então clique em “Select an Assembly” selecione a pasta “C:\Arquivos de programas\FirebirdNETProvider1.7\” e então selecione o arquivo: “FirebirdSql.Data.Firebird.dll”

 

Imagem 2

 

Clique em abrir e então clique em ok, pronto nosso driver está integrado ao design time do delphi, você pode conferir criando uma nova aplicação Winforms e verificando na toolpalete 4 componentes na seção “Firebird”, são eles: FBCommand, FBCommandBuilder, FBConnection e FBDataAdapter.

 

Imagem 3

 

·       Primeiros passos

 

Agora que instalamos e integramos o driver ao delphi já estamos aptos a começar, vou ensinar agora como usar o FBConnection, conectar-se a base de dados, utilizar transações, extração de metadata e outros.

 

Componente FBConnection

 

·        Criando a primeira Connectionstring:

 

Crie um novo projeto WinForms Application e no formulário coloque um FBConnection, note na object inspector a propriedade “ConnectionString” clique no botão desta propriedade e a tela a seguir será exibida:

 

Imagem 4

 

1.      O campo “Data Source” deve ser preenchido com o endereço IP onde seu servidor de banco de dados se encontra instalado, no meu caso o servidor está na própria máquina eu poderia deixar em branco ou digitar localhost;

2.      O campo “Data Source Port” serve para indicar em qual porta o seu servidor de banco de dados está operando, a porta padrão é a 3050;

3.      Em “Dialect” você deve escolher o Dialeto da conexão com o banco de dados, o dialeto padrão é o 3;

4.      Em “Charset” você deve selecionar o Charset do seu banco de dados, o Charset padrão é o NONE;

5.      O campo Database, é o caminho completo para a base de dados, ex: C:\Arquivos de programas\Firebird\Firebird_1_5\examples\EMPLOYEE.FDB

6.      Em login coloque o usuário e senha do seu banco de dados, padrão user = SYSDBA e password = masterkey, você pode definir também uma Role pra esse login ou simplesmente deixar o campo Role em branco.

 

Em “Connection Settings” você encontrará o sistema de connections pooling, se estiver habilitado, a conexão é mantida em memória, assim se você fechar o FBConnection e abrir denovo ele irá reaproveitar esta conexão, tornando mais rápido o processo. Você pode definir o tempo que a conexão ficará armazenada no pool após ser fechada no campo Life Time, se deixar como 0 a conexão ficará no pool até o objeto connection ser destruído. Neste grupo você também encontra a propriedade timeout que é o tempo que o FBConnection irá tentar se conectar antes de retornar um erro (caso a conexão não seja realizada com sucesso). O Packet size serve para indicar o tamanho dos pacotes que serão transmitidos na conexão com o servidor, e por último o Server Type que serve para definir se seu aplicativo vai se conectar a servidor Super ou Classic ou a servidor Embeded.

 

Agora que sabemos pra que serve cada um dos elementos da connectionstring editor vamos nos conectar a um banco de testes, você pode usar o banco de dados de exemplo que vem com o firebird: C:\Arquivos de programas\Firebird\Firebird_1_5\examples\ EMPLOYEE.FDB, copie ele para uma outra pasta para que você tenha sempre uma cópia limpa dele para testar outros exemplos, vamos moldar a connectionstring da seguinte maneira:

 

Imagem 5

 

Lembre-se de substituir o caminho C:\ClubeDelphi\Firebird .NET pelo caminho para onde você copiou o banco de dados EMPLOYE.FDB. Clique em “Test” e veja se a mensagem Connection successful aparece, se sim clique em Accept, veja que agora na propriedade connectionstring está preenchida da seguinte forma:

 

User=SYSDBA;Password=masterkey;Database=C:\ClubeDelphi\Firebird.NET\EMPLOYEE.FDB;
DataSource=;Port=3050;Dialect=3;Charset=NONE;Role=;Connection lifetime=0;
Connection timeout=15;Pooling=True;Packet Size=8192;Server Type=0

 

Com base nessa string você pode alterar a conexão via código. Veja um exemplo:

 

function BuildConnectionString(Database, Servidor, User, Pass: System.String): String;

begin

Result := User=+User.ToUpper+;+

Password=+Pass+;+

Database=+Database+;+

DataSource=+Servidor+;+

Port=3050;+

Dialect=3;+

Charset=NONE;+

Role=;+

Connection lifetime=0;+

Connection timeout=15;+

Pooling=False;+

Packet Size=8192;+

Server Type=0;;

end;

 

Usaríamos assim: FbConnection1.ConnectionString := BuildConnectionString(C:\ClubeDelphi\Firebird .NET\ EMPLOYEE.FDB,localhost,SYSDBA,masterkey);

 

ou ainda usar a classe FbConnectionStringBuilder:

function BuildConnectionString(ArquivoFDB: System.String): System.String;

var

FbConnectionStB: FirebirdSQL.Data.Firebird.FbConnectionStringBuilder;

begin

FbConnectionStB := FirebirdSQL.Data.Firebird.FbConnectionStringBuilder.Create;

try

FbConnectionStB.UserID := SYSDBA;

FbConnectionStB.Password := masterkey;

FbConnectionStB.DataSource := localhost;

FbConnectionStB.Database := ArquivoFDB;

FbConnectionStB.Port := 3050;

FbConnectionStB.Dialect := 3;

FbConnectionStB.Charset := NONE;

FbConnectionStB.ConnectionLifeTime := 0;

FbConnectionStB.ConnectionTimeout := 15;

FbConnectionStB.Pooling := True;

FbConnectionStB.PacketSize := 8192;

FbConnectionStB.ServerType := 0;

Result := FbConnectionStB.ConnectionString;

finally

FbConnectionStB.Free;

end;

end;

 

Para usar faça assim:

FBConnection1.ConnectionString := BuildConnectionString(C:\ClubeDelphi\Firebird .NET\ EMPLOYEE.FDB);

 

Note que isso é só uma demonstração de utilização, um exemplo de como é fácil usar estas funções para configurar diferentes FBConnections sem necessidade de repetição de código, existirão também diversas situações que você irá precisar usar o FbConnectionStringBuilder de outra forma.

Vamos ver agora os principais métodos do FbConnection, para que servem e como aplicá-los:

1.      CreateDatabase – Cria um novo banco de dados de acordo com a connectionstring passada.

2.      DropDatabase – Apaga um banco de dados especificado na connectionstring passada.

3.      BeginTransaction – Inicia uma transação retornando um objeto FBTransaction.

4.      ChangeDatabase – Alterna a conexão do banco de dados para outro arquivo.

5.      Close – Fecha a conexão ativa.

6.      CreateCommand – Cria um objeto FBCommand já com sua conexão apontada para o objeto FBConnection

7.      GetSchema – Serve para retornar os metadados, incluindo também objetos de sistema do banco.

8.      Open – Abre a conexão com o banco de dados.

 

Exemplos de utilização dos métodos

 

1.      CreateDatabase: 

Ex: FBConnection1.CreateDatabase(User=SYSDBA;Password=masterkey;
Database=C:\ClubeDelphi\Firebird.NET\TESTE_CREATE_DATABASE.FDB;
DataSource=;Port=3050;Dialect=3;Charset=NONE;Role=;Connection lifetime=0;
Connection timeout=15;Pooling=True;Packet Size=8192;Server Type=0;);

 

Além da connectionstring podemos informar ainda:

PageSize, é o tamanho da página do banco.

ForcedWrites, define se o banco vai ser criado forçando escrita.

Overwrite, se estiver true sobrescreve o banco de dados existente passado como parâmetro na connectionstring.

 

2.      DropDatabase:

 

Ex: FBConnection1.DropDatabase(User=SYSDBA;Password=masterkey;
Database=C:\ClubeDelphi\Firebird.NET\TESTE_CREATE_DATABASE.FDB;
DataSource=;Port=3050;Dialect=3;Charset=NONE;Role=;Connection lifetime=0;
Connection timeout=15;Pooling=True;Packet Size=8192;Server Type=0;);

 

Como você deve ter notado esse método é extremamente perigoso e acho que nem deveria existir, a exclusão de um arquivo tão importante como um banco de dados é um ato que requer muita atenção e cuidado, use-o com extrema cautela e sempre se certifique que exista um backup.

Nota: Tanto o método CreateDatabase quanto o DropDatabase são considerados obsoletos e deverão ser excluídos nas próximas versões do provider.

 

3.      BeginTransaction:

Esta função serve para que você inicie uma nova transação a qual é retornada como um objeto da classe FBTransaction, veja um exemplo prático:

 

FBCommand1.Transaction := FBConnection1.BeginTransaction;

 

Você pode especificar opções para o início da transação, através do objeto FbTransactionOptions, veja as opções disponíveis:

  • Consistency
  • Concurrency
  • Shared
  • Protected
  • Exclusive
  • Wait
  • NoWait
  • Read
  • Write
  • LockRead
  • LockWrite
  • ReadCommitted
  • Autocommit
  • RecVersion
  • NoRecVersion
  • RestartRequests
  • NoAutoUndo

Nota: Repare na quantidade de opções disponíveis para o sistema de transações, isso abre um leque de possibilidades para suas aplicações. 

Ex: FBCommand1.Transaction := FBConnection1.BeginTransaction(FirebirdSQL.Data.Firebird.FbTransactionOptions.ReadCommitted);

Você pode também definir o nível de isolamento através do parâmetro isolationlevel, veja os tipos de isolamento:

  • Chaos
  • ReadCommitted
  • ReadUncommitted
  • RepeatableRead
  • Serializable
  • Unspecified

 

Ex: FBCommand1.Transaction := FbConnection1.BeginTransaction(System.Data.IsolationLevel.ReadCommitted); 

Além destas opções pode-se usar o BeginTransaction acrescentando ainda um nome para a transação:

Ex: FBCommand1.Transaction := FBConnection1.BeginTransaction(Minha_Transacao);

Você pode passar este nome como segundo parâmetro se precisar usar FbTransactionOptions ou mesmo IsolationLevel:

 

Ex: FBCommand1.Transaction := FbConnection1.BeginTransaction(System.Data.IsolationLevel.ReadCommitted, Minha_Transacao);

 

Para encerrar uma transação utilize Transaction.RollBack para desfazer as alterações efetuadas, ou Transaction.Commit para confirmar as alterações.

 

Um detalhe importante é que ao iniciar uma transação esta deve ser finalizada antes que uma nova transação seja iniciada, o driver não suporta duas ou mais transações em paralelo para o mesmo FBConnection.

 

4.      ChangeDatabase:

Alterna para outro arquivo de banco de dados, mudando a propriedade Database do componente FBConnection, se o FBConnection estiver aberto esta função irá fecha-lo e tentará abrir o arquivo passado como parâmetro, usando os outros parâmetros da ConnectionString pré-configurada.

 

Ex: FBConnection1.ChangeDatabase(C:\ClubeDelphi\Firebird .NET\TESTE.FDB);

5.      Close:

Fecha a conexão com o banco de dados. Sempre utilize este método para encerrar a conexão ou o arquivo ficará amarrado ao processo, supondo-se que você tenha uma tela onde uma conexão é aberta, ao fechar esta tela você deve encerrar a conexão, pois apesar do objeto FBConnection ser destruído ele não se desconecta automaticamente, a desconexão então só seria realizada quando o aplicativo fosse finalizado.

 

6.      CreateCommand:

Serve para criar um FBCommand já ligado ao FBConnection em tempo de execução.

 

Ex: var MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;
begin
MeuFBCommand := FBConnection1.CreateCommand;
MeuFBCommand.Transaction := FBConnection1.BeginTransaction(Minha_Transacao);
MeuFBCommand.CommandText := UPDATE CLIENTES SET CPF = 1234567890 WHERE ID = 250;
MeuFBCommand.ExecuteNonQuery();

end;

 

7.      GetSchema:


Extraindo informações do banco de dados utilizando o FBConnection:

Através da função GetSchema() do FBConnection podemos retornar a metadata e também objetos de sistema do banco de dados, ou seja podemos extrair nomes de tabelas e seus campos, índices, triggers e tudo mais que houver no banco: a função GetSchema pode ser usada sem parâmetros (retorna uma lista com os itens disponíveis para extração), com um parâmetro do tipo System.String (retorna somente o que foi especificado), com uma array do objeto System.String (retorna os itens que forem especificados em cada elemento da array) e ainda com um segundo parâmetro do tipo array de strings onde você aplica restrições as informações retornadas. Veja a lista de parâmetros possíveis:

 

1.            CharacterSets – Retorna os charsets.

2.            CheckConstraints – Retorna as checks constraints.

3.            CheckConstraintsByTable – Retorna as checks constraints por tabela.

4.            Collations – Retorna os collates.

5.            Columns – Retorna os campos das tabelas (incluindo views).

6.            ColumnPrivileges – Retorna os privilégios dos campos.

7.            DataTypes – Retorna os tipos de campos suportados pelo firebird.

8.            Domains – Retorna os domínios definidos no banco.

9.            ForeignKeys – Retorna as chaves estrangeiras.

10.        Functions – Retorna as funções externas, ou seja, as UDFs instaladas no banco de dados.

11.        Generators – Retorna os generators.

12.        Indexes – Retorna os índices.

13.        MetaDataCollections – Retorna uma lista com a metadata suportada. O mesmo que usar GetSchema sem parâmetros.

14.        PrimaryKeys – Retorna as chaves primárias.

15.        ProcedureParameters – Retorna todos os parâmetros de storedprocedures.

16.        ProcedurePrivileges – Retorna os privilégios das storedprocedures.

17.        Procedures – Retorna as storedprocedures.

18.        Restrictions – Retorna uma lista com as restrições de extração suportada por cada tipo de metadata. Essa opção é usada como segundo parâmetro para restringir os itens que serão retornados com o GetSchema.

19.        Roles – Retorna as roles.

20.        TableConstraints – Retorna todas as contraints das tabelas, incluindo índices, checks e outros.

21.        TablePrivileges – Retorna os privilégios da tabela.

22.        Tables – Retorna todas as tabelas, não retorna views.

23.        Triggers – Retorna todas as triggers do banco.

24.        UniqueKeys – Retorna todas as chaves únicas do banco.

25.        ViewColumnUsage – Retorna a relação de campos utilizados em views.

26.        ViewPrivileges – Retorna os privilégios das views.

27.        Views – Retorna as views.

Para exibir os dados da metadata numa datagrid basta ligar o resultado da função GetSchema a propriedade datasource, veja um exemplo:

Datagrid1.DataSource := FBConnection1.GetSchema (Tables);

ou

Datagrid1.DataSource := FBConnection1.GetSchema (Tables).DataSet.Tables[0];

 

Como você deve ter notado o resultado do GetSchema retorna todas as informações possíveis, incluindo objetos de sistema do banco, se quisermos por exemplo retornar somente as tabelas do banco precisaremos filtrar esse resultado, para fazer isso iremos usar o segundo parâmetro do GetSchema, a opção Restrictions. As Restrictions para Tables são: Catalog, Schema, Name e Type. Para filtrar somente as tabelas do usuário vamos tomar como base o seguinte código:

DataGrid1.DataSource := FBConnection1.GetSchema(Tables, [nil, nil, nil, TABLE]);

 

Onde Catalog = nil Schema = nil Name = nil Type = TABLE, ou seja, Catalog = sem filtro, Schema = sem filtro, Name = sem filtro e Type = somente o que for igual a TABLE. Lembrando que para descobrir quais são as Restrictions do elemento que você precisa retornar use GetSchema(Restrictions);

 

  1. Open:

    Em alguns casos é acionado automaticamente como quando fazemos uma consulta usando o FBDataAdapter e após a consulta ser realizada o FBConnection é fechado, porém para realizar algumas operações como o GetSchema por exemplo teremos que fazer isso manualmente, é recomendável, para tornar o aplicativo mais otimizado, fechar a conexão imediatamente após terminar de usá-la, usar um try finally é uma boa idéia, criar a conexão em tempo de execução e destruí-la logo em seguida também é uma boa idéia, especialmente se estiver trabalhando com ASP.NET.

 

Vamos agora ver algumas propriedades e mostrar alguns exemplos.

 

1.      ConnectionString – Armazena a string de conexão com o banco de dados.

2.      ConnectionTimeOut – Tempo da tentativa de conexão, propriedade somente leitura, é preenchida quando se altera a connectionstring;

3.      Database – Caminho do arquivo do banco de dados, propriedade somente leitura, é preenchida quando se altera a connectionstring;

4.      DataSource – IP do servidor de banco de dados, propriedade somente leitura, é preenchida quando se altera a connectionstring;

5.      PacketSize – Tamanho do pacote em byte que é usado para transmissão dos dados na conexão, propriedade somente leitura, é preenchida quando se altera a connectionstring;

6.      ServerVersion – Propriedade somente leitura que retorna a versão do servidor do banco de dados onde o FBConnection está conectado, a conexão tem que estar aberta para ser usado.

7.      State – Retorna o status do FBConnection, eles podem ser:

1)      Broken – Ocorre quando a conexão foi interrompida de forma irregular.

2)      Closed – A conexão está fechada.

3)      Connecting – A conexão está sendo efetuada.

4)      Executing – Algum comando está sendo executado.

5)      Fetching – O FBConnection está retornando registros.

6)      Open – A conexão está ativa.

 

Você pode testar o estado do seu FBConnection da seguinte forma:

 

Ex:
case FBConnection1.State of

ConnectionState.Closed: MessageBox.Show(ConnectionState.Closed);

ConnectionState.Open: MessageBox.Show(ConnectionState.Open);

ConnectionState.Connecting: MessageBox.Show(ConnectionState.Connecting);

ConnectionState.Executing: MessageBox.Show(ConnectionState.Executing);

ConnectionState.Fetching: MessageBox.Show(ConnectionState.Fetching);

ConnectionState.Broken: MessageBox.Show(ConnectionState.Broken);

end;

 

O FBConnection possui apenas dois eventos, são eles:

 

1)      InfoMessage – Ocorre quando o servidor envia alguma mensagem de alerta ao cliente.

2)      StateChange – Ocorre quando o State é alterado.

 

Exemplo InfoMessage:

 

procedure TWinForm1.FbConnection1_InfoMessage(sender: System.Object; e: FirebirdSql.Data.Firebird.FbInfoMessageEventArgs);

begin

MessageBox.Show(e.Message);

end;

 

Exemplo StateChange:

procedure TWinForm1.FbConnection1_StateChange(sender: System.Object; e: System.Data.StateChangeEventArgs);

var

EstadoOriginal,

EstadoAtual: System.String;

begin

case e.OriginalState of

ConnectionState.Closed: EstadoOriginal := ConnectionState.Closed;

ConnectionState.Open: EstadoOriginal := ConnectionState.Open;

ConnectionState.Connecting: EstadoOriginal := ConnectionState.Connecting;

ConnectionState.Executing: EstadoOriginal := ConnectionState.Executing;

ConnectionState.Fetching: EstadoOriginal := ConnectionState.Fetching;

ConnectionState.Broken: EstadoOriginal := ConnectionState.Broken;

end;

case e.CurrentState of

ConnectionState.Closed: EstadoAtual := ConnectionState.Closed;

ConnectionState.Open: EstadoAtual := ConnectionState.Open;

ConnectionState.Connecting: EstadoAtual := ConnectionState.Connecting;

ConnectionState.Executing: EstadoAtual := ConnectionState.Executing;

ConnectionState.Fetching: EstadoAtual := ConnectionState.Fetching;

ConnectionState.Broken: EstadoAtual := ConnectionState.Broken;

end;

MessageBox.Show(Estado original: + EstadoOriginal + #13#10 +

Estado atual: + EstadoAtual);

end;

 

 

Componente FBCommand

 

O componente FBCommand é o responsável por executar comandos SQL no banco de dados e também pode ser usado para realizar consultas que retornem registros porém não pode ser ligado componentes de tela para edição dos registros, vamos ver agora suas principais características:

 

·        Criando um novo FBCommand:

 

A construção deste componente pode ocorrer de diferentes formas, você pode simplesmente dropá-lo no form em designtime, pode usar o FBConnection.CreateCommand passando o resultado desta função para uma variável do tipo FBCommand ou mesmo chamar diretamente seu constructor e cria-lo dinâmicamente, neste último caso você pode passar o comando que deseja executar, FBConnection e FBTransaction, veja alguns exemplos:

 

var

MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;

begin

MeuFBCommand := FBConnection1.CreateCommand;

(…)

 

Ou

 

var

MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;

begin

MeuFBCommand := FirebirdSQL.Data.Firebird.FBCommand.Create(SELECT * FROM CLIENTES);

MeuFBCommand.Transaction := FBConnection1.BeginTransaction;

MeuFBCommand.Connection := FBConnection1;

(…)

 

Ou

 

var

MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;

begin

MeuFBCommand := FirebirdSQL.Data.Firebird.FBCommand.Create(SELECT * FROM CLIENTES, FBConnection1);

MeuFBCommand.Transaction := FBConnection1.BeginTransaction;

(…)

 

Ou

 

var

MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;

begin

MeuFBCommand := FirebirdSQL.Data.Firebird.FBCommand.Create(SELECT * FROM CLIENTES, FBConnection1, FBConnection1.BeginTransaction);

(…)

 

 

Repare que a última opção é bem mais prática de se trabalhar, com apenas uma linha o FBCommand está pronto para ser executado.

 

Principais métodos do FBCommand:

 

1.      Cancel – Tenta cancelar a execução do comando.

2.      CreateParameter – Cria um novo FbParameter.

3.      ExecuteNonQuery – Serve para executar INSERT, UPDATE, DELETE, CREATE e outros, retorna a quantidade de registros afetados pelo comando, não retorna linhas de registros.

4.      ExecuteReader – Serve para fazer consultas do tipo SELECT no banco de dados, retorna um objeto do tipo FbDataReader, você pode passar flags do tipo CommandBehavior como parâmetro.

5.      ExecuteScalar – Também para consultas SELECT, mas só traz o valor da primeira coluna do primeiro registro, os outros registros e colunas são ignorados

6.      Prepare – Cria uma versão compilada do CommandText, prepare é chamado automaticamente ao se executar ExecuteNonQuery, ExecuteReader e ExecuteScalar.

 

Exemplos de utilização dos métodos:

 

1.      Cancel: Usado para cancelar a execução do comando, é uma implementação para o .NET provider do firebird do IDbCommand.Cancel.

 

2.      CreateParameter: Retorna um objeto do tipo FbParameter. (Mais a frente irei detalhar sua utilização).

3.      ExecuteNonQuery: Usado para executar comandos que não retornem registro. Retorna a quantidade de registros afetados pelo comando.

Ex: var
MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;
begin
MeuFBCommand := FirebirdSQL.Data.Firebird.FBCommand.Create(UPDATE “COUNTRY” SET COUNTRY = TESTE WHERE COUNTRY = FRANCE , FBConnection1, FBConnection1.BeginTransaction);
if MeuFBCommand.ExecuteNonQuery = 1 then
MeuFBCommand.Transaction.Commit;
(…)

4.      ExecuteReader: Usado para fazer consultas, retorna um objeto do tipo FBDataReader. Um detalhe importante é que o FBConnection fica ocupado, não permitindo outras operações enquanto o FBDataReader estiver aberto, portanto ao terminar use Close. Lembrando também que o FBDataReader é unidirecional. Logo à frente irei detalhar sobre esse objeto.

Ex: var
MeuFBCommand: FirebirdSQL.Data.Firebird.FBCommand;
MeuFBDataReader: FirebirdSQL.Data.Firebird.FBDataReader;
begin
MeuFBCommand := FirebirdSQL.Data.Firebird.FBCommand.Create(SELECT FIRST_NAME FROM EMPLOYEE WHERE JOB_COUNTRY = USA , FBConnection1, FBConnection1.BeginTransaction);
MeuFBDataReader := MeuFBCommand.ExecuteReader;
While (MeuFBDataReader.Read) do
TBEmployeeByCountry.Lines.Add(MeuFBDataReader.GetString(1));
MeuFBDataReader.Close;
end;

5.      ExecuteScalar: Essa função também é usada para fazer consultas, porém só retorna a primeira coluna da primeira linha, todo o resto é ignorado, é ideal para usar com consultas de contagem e soma (valores agregados em geral), veja os exemplos:

Ex: var
QuantidadeRegistros: System.Int32;
begin
QuantidadeRegistros := FirebirdSQL.Data.Firebird.FBCommand.Create(SELECT COUNT(*) FROM EMPLOYEE WHERE JOB_COUNTRY = USA , FBConnection1, FBConnection1.BeginTransaction).ExecuteScalar;
end;

6.      Prepare: Cria uma versão compilada do seu SQL. Não é necessário usar esta função, ela é chamada automaticamente ao se executar ExecuteNonQuery, ExecuteReader e ExecuteScalar.

Ex:
var

MeuFBCmd: FirebirdSQL.Data.Firebird.FbCommand;

begin

MeuFBCmd := FirebirdSQL.Data.Firebird.FbCommand.Create(UPDATE CLIENTES SET NOME = FULANO WHERE ID = 55, FBConnection1, FBConnection1.BeginTransaction);

MeuFBCmd.Prepare;
(...)

 

Vamos agora ver algumas propriedades do FBCommand:

 

1.      CommandPlan – Retorna a versão compilada do CommandText

2.      CommandText – Comando a ser executado no banco de dados

3.      Connection – O FBConnection utilizado pelo componente FBCommand.

4.      Parameters – Coleção de parâmetros do FBCommand.

5.      Transaction – O objeto FBTransaction utilizado pelo componente FBCommand.

 

Exemplos de utilização dos métodos:

 

1.      CommandPlan: Retorna o plan usado pelo servidor para executar o comando.

2.      CommandText: Comando SQL a ser executado.

3.      Connection: FBConnection utilizado pelo componente, deve ser apontado antes de qualquer operação com o FBCommand.

4.      Parameters: É uma coleção de objetos do tipo FbParameter, você pode realizar consultas ou mesmo fazer alterações com a ajuda desses parâmetros, é bem semelhante aos parâmetros do TDataset no Delphi.

Ex: MeuFBCmd.Parameters.Add(@JOB_COUNTRY, FbDbType.VarChar, 15).Value := USA;

no comando sql use assim: SELECT * FROM EMPLOYEE WHERE JOB_COUNTRY = @JOB_COUNTRY

Nota: os parâmetros devem possuir o símbolo @ antes do nome.

5.      Transaction: A transação deve ser passada ao componente FBCommand antes de executar qualquer operação, lembre-se também de ativa-la.

Ex: Vide “Criando um novo FBCommand” ou “FBConnection.BeginTransaction”


O FBCommand não possui eventos.

Componente FBCommandBuilder


O FBCommandBuilder é o componente usado para gerar automaticamente as cláusulas de INSERT, UPDATE e DELETE, o componente é bem simples e sua forma de usar também, vamos ver seus principais métodos:

 

1.      GetDeleteCommand – Retorna um componente FBCommand configurado com o SQL de DELETE.

2.      GetInsertCommand – Retorna um componente FBCommand configurado com o SQL de INSERT.

3.      GetUpdateCommand – Retorna um componente FBCommand configurado com o SQL de UPDATE.

4.      RefreshSchema – Atualiza as informações de campos para gerar os comandos SQL de INSERT, UPDATE e DELETE, deve ser executa sempre que a cláusula SELECT do FBDataAdapter for modificada.

Exemplos de utilização:

 

GetDeleteCommand, GetInsertCommand e GetUpdateCommand: Ao apagar, alterar ou incluir um registro usando o FBDataAdapter + DataSet + DataTable esses procedimentos são feitos “OffLine” ou seja as alterações não são executadas diretamente no banco de dados e sim em memória, essa maneira de trabalhar é uma implementação nativa do .NET Framework, note que há uma certa semelhança com ClientDataSet + Provider + DataSet onde as alterações eram feitas em memória no ClientDataSet e executas pela função ApplyUpdates do próprio ClientDataSet, no caso do FBDataProvider é necessário que seja passada a cláusula de alteração pro banco de dados através dos SubComponentes DeleteCommand, UpdateCommand e InsertCommand do FBDataAdapter com o FBCommandBuilder esses SubComponentes são criados automaticamente já com as cláusulas de alteração, exclusão e inserção de registros. Veja:

 

Ex: Após realizar uma alteração diretamente no datagrid você terá que propagar a alteração para o banco de dados:

FBConnection1.Open;
//Sempre use o FBCommandBuilder para gerar a cláusula de update, delete ou insert antes de usar o commando Update do

//do FBDataAdapter

FBDataAdapter1.UpdateCommand := FbCommandBuilder1.GetUpdateCommand;

FBDataAdapter1.Update(DS_EMPLOYEE,EMPLOYEE);

FBConnection1.Close;

 

O mesmo esquema serve também para DELETE e INSERT:

FBDataAdapter1.DeleteCommand := FbCommandBuilder1.GetDeleteCommand; e
FBDataAdapter1.InsertCommand := FbCommandBuilder1.GetInsertCommand;

 

Nota: Lembre-se de apontar a propriedade DataAdapter do FBCommandBuilder para o FBDataAdapter com o qual você estiver trabalhando.

O FBDataAdapter e o FBCommandBuilder trabalham da mesma forma que os componente SQLDataAdapter e o SQLCommandBuilder, se quiser se aprofundar mais sobre sua utilização procure na documentação do .NET FrameWork sobre estas classes.

RefreshSchema: Essa função deve ser sempre chamada após alterar o código SQL do select, é ela que informa ao FBCommandBuilder quais campos estão disponíveis para alterações.

 

Ex: FBDataAdapter1.SelectCommand.CommandText := SELECT * FROM EMPLOYEE;
FBDataAdapter1.Fill(DS_EMPLOYEE, EMPLOYEE);

FBCommandBuilder1.RefreshSchema;

Principais propriedades:

 

1.      CommandBuilderBehavior – Opção que define que tipo de campos serão usados para gerar as cláusulas WHERE.

2.      DataAdapter – O componente FBDataAdapter que fará as alterações no banco.

3.      QuotePrefix – O delimitador que será usado para identificar as Tabelas e Campos no SQL gerado. Delimitador da esquerda.

4.      QuoteSufix – O delimitador que será usado para identificar as Tabelas e Campos no SQL gerado. Delimitador da direita.

5.      TimestampColumnName – Usado jutamente com a propriedade CommandBuilderBehavior, nomes de um campo do tipo TIMESTAMP para ser usado como chave nas alterações.

Exemplos de utilização:

1.      CommandBuilderBehavior: Usado para definir a cláusula WHERE dos comandos gerados, opções disponíveis:

1)      Default: igual a AllFields

2)      AllFields: Gera o SQL constando na cláusula where a todos os campos disponíveis. Ex: UPDATE TABELA SET NOME = FULANO WHERE (SOBRENOME = BELTRANO) AND (ENDERECO = RUA DAS OLIVEIRAS) AND (CEP = 40000-000);

3)      KeyFields: Gera o SQL constando na cláusula where somente os campos que são chaves.Ex: UPDATE TABELA SET NOME = FULANO WHERE (ID = 5);

4)      KeyAndTimestampFields: Gera o SQL constando na cláusula where os campos chaves mais o campo especificado em TimeStampColumnName. Ex: UPDATE TABELA SET NOME = FULANO WHERE (ID = 5) AND (MEU_CAMPO = ?????);

2.      DataAdapter: o FBCommandBuilder deve ser apontado para um FBDataAdapter antes de chamar as funções GetXXXXCommand;

3.      QuotePrefix: o delimitador inicial para nomes de Tabelas e Campos: Ex: se o delimitador inicial(QuotePrefix) e o delimitador final(QuoteSufix) forem aspas(“) o sql ficaria assim: UPDATE “TABELA” SET “CAMPO” = Valor;

4.      QuoteSufix: delimitador final para nomes de Tabelas e Campos.

5.      TimestampColumnName: Nome do campo que será usado como chave da cláusula WHERE juntamente com as chaves da tabela, veja exemplo “4” em CommandBuilderBehavior logo acima.

 

O FBCommandBuilder não possui eventos.

Vimos que o FBCommandBuilder facilita nosso trabalho fazendo a árdua tarefa de pegar tudo que foi alterado no DataTable e propagar automaticamente para o FBDataAdapter, veremos agora como manipular usuários e outros serviços através da classe FBServices.

FirebirdSql.Data.Firebird.Services

Para finalizar nosso artigo veremos agora alguns exemplos de como utilizar algumas das classes que implementam os serviços do firebird como gerenciamento de usuários, backup e restore.

Para todos os exemplos abaixo você deve declarar o namespace: FirebirdSQL.Data.Firebird.Services na seção USES.

 

1.      Classe FbBackup – É com esta classe que você efetua backups de sua base dados, veja as principais propriedades:

1)      BackupFiles: Array onde você pode especificar os arquivos de backup, utiliza a classe FBBackupFile.

2)      ConnectionString: String de conexão com o banco de dados que será feita a cópia de segurança, veja ConnectionString em FBConnection.

3)      Options: Opções para realização do backup, enumeration FbBackupFlags.

4)      State: Retorna o estado do serviço, enumeration FbServiceState, pode ser Open ou Closed.

5)      Verbose: Se estive true mostra o que o FBBackup está fazendo, para capturar o verbose use o evento ServiceOutput.

Para executar o backup use a função Execute. Abaixo um exemplo completo de como realizar o backup e capturando o evento ServiceOutput, tudo via código:

Ex:

//mostra o verbose do backup em um TextBox de múltiplas linhas, serve tanto para backup quanto para o restore

procedure TWinForm1. OnBackupRestoreBD (sender: System.Object; e: ServiceOutputEventArgs);

begin

Application.DoEvents;

MeuTextBox.AppendText(#13#10 + e.Message);

end;


procedure TWinForm1.BackupDB(FBConn: FirebirdSQL.Data.Firebird.FBConnection);

var

FBBack: FirebirdSql.Data.Firebird.Services.FbBackup;

ArquivoSaida: FirebirdSql.Data.Firebird.Services.FbBackupFile;

NomeArquivo: System.String;

begin

if FbConn.State = System.Data.ConnectionState.Closed then

raise Exception.Create(Banco de dados desconectado!);

//

FBBack := FirebirdSql.Data.Firebird.Services.FbBackup.Create;

try

//captura o evento ServiceOutput para mostrar o verbose do backup

Include(FBBack.ServiceOutput, OnBackupBD);

//como o FbConn já está conectado, pega sua connectionstring

FBBack.ConnectionString := FbConn.ConnectionString;

FBBack.Verbose := True;

//cria o diretório de backup

if System.IO.Directory.Exists(Application.get_StartupPath+\Backups\) = False then

System.IO.Directory.CreateDirectory(Application.get_StartupPath+\Backups\);

//gera o nome do arquivo

NomeArquivo := Application.get_StartupPath+\Backups\bak_ +

Path.GetFileName(FBConn.Database) + .fbk;

//o firebird.net backup service suporta vários arquivos de saída listados

//em uma array, neste caso iremos usar apenas um arquivo

ArquivoSaida := FirebirdSql.Data.Firebird.Services.FbBackupFile.Create(NomeArquivo, -1);

try

FBBack.BackupFiles.Add(ArquivoSaida);

FBBack.Execute;

finally

ArquivoSaida.Free;

end;

finally

FBBack.Free;

end;

end;

Para usar chame BackupDB(FBConnection1) por exemplo, ele irá criar o backup na pasta Backups dentro da pasta do seu executável.

 

2.      Classe FBRestore – Restaura um backup previamente realizado, veja as principais propriedades:

1)      BackupFiles: Array onde você pode especificar os arquivos de backup, utiliza a classe FBBackupFile.

2)      ConnectionString: String de conexão com o banco de dados que será backupeado, veja ConnectionString em FBConnection.

3)      Options: Opções para realização da restauração, enumeration FbRestoreFlags.

4)      PageSize: Tamanho da página do banco restaurado, pode ser: 1024, 2048, 4096, 8192 ou 16384 bytes.

5)      State: Retorna o estado do serviço, enumeration FbServiceState, pode ser Open ou Closed.

6)      Verbose: Se estive true mostra o que o FBBackup está fazendo, para capturar o verbose use o evento ServiceOutput.


Para executar o restore use a função Execute. Abaixo um exemplo completo de como realizar o restore e capturando o evento ServiceOutput, tudo via código:

Ex:

procedure TWinForm1.RestoreDB(FBConn: FirebirdSQL.Data.Firebird.FBConnection;

3.      Classe FBSecurity – Implementa listagem, exclusão, inserção e modificação dos usuários do servidor do banco de dados.

Principais propriedades:

1)      ConnectionString: String de conexão com o banco de dados, veja ConnectionString em FBConnection.

2)      State: Retorna o estado do serviço, enumeration FbServiceState, pode ser Open ou Closed.

3)      UsersDbPath: Caminho para o banco de dados de segurança (Security.fdb) no servidor do banco de dados selecionado. Ex: “C:\Meu Sistema\Firebird\” ou “/usr/firebird” caso o servidor seja linux. Principais métodos:

1)      AddUser: Serve para adicionar um novo usuário ao banco de dados, necessita de um parâmetro do tipo FBUserData.

2)      DeleteUser: Serve para apagar um usuário do banco de dados, necessita de um parâmetro do tipo FBUserData para encontrar o usuário a ser removido.

3)      DisplayUser: Retorna todos os dados de um usuário através da classe FBUserData, necessita que se passe o UserName deste usuário como parâmetro.

4)      DisplayUsers: Retorna uma array da classe FBUserData com os dados de todos os usuários do banco de dados de segurança.

5)      ModifyUser: Altera as informações do usuário selecionado, necessita que se passe os dados do usuário através de um objeto da classe FBUserData.

 

Veja abaixo um exemplo completo de como criar um usuário:

procedure CriaUsuario(FBConn: FirebirdSQL.Data.Firebird.FBConnection;

UserName, Senha: System.String);

var

FbSec: FirebirdSql.Data.Firebird.Services.FbSecurity;

UserD: FirebirdSql.Data.Firebird.Services.FbUserData;

Users: System.Array;

I: System.Int32;

begin

if FbConn.State = System.Data.ConnectionState.Closed then

raise Exception.Create(Banco de dados desconectado!);

UserD := FirebirdSql.Data.Firebird.Services.FbUserData.Create;

FbSec := FirebirdSql.Data.Firebird.Services.FbSecurity.Create;

try

//verifica se o usuário existe

Users := FbSec.DisplayUsers;

for I := 0 to Users.Length - 1 do

if (Users[I] as FirebirdSql.Data.Firebird.Services.FbUserData).UserName = UserName then

Exit;

 

UserD.UserName := UserName;

UserD.UserPassword := Senha;

 

FbSec.AddUser(UserD);

finally

UserD.Free;

FbSec.Free;

end;

end;

4.      Classe FBUserData – Usado em conjunto com o FBSecurity, armazena os dados do usuário.

Lista de propriedades:

1)      FirstName – Primeiro nome do usuário.

2)      GroupID – ID do grupo de usuários a qual o usuário pertence.

3)      GroupName – Nome do grupo a qual ele pertence.

4)      LastName – Último do usuário.

5)      MiddleName – Nome do meio.

6)      RoleName – Nome da Role que será aplicada a este usuário.

7)      UserID – ID do usuário.

8)      UserName – Nome do usuário.

9)      UserPassword – Senha do usuário.

Ex: MeuUsuario := FBUserData.Create;
MeuUsuario.FirstName := JOSÉ;

MeuUsuario.MiddleName := ALBERTO;
MeuUsuario.LastName := SILVA;
MeuUsuario.UserName := JOSE_ALBERTO;
MeuUsuario.UserPassword := 123456;

 

Por aqui encerramos nosso artigo, vimos que o Firebird .NET Provider é muito mais do que uma opção de acesso a esse sensacional banco de dados, esse driver simplesmente reuniu todas as funcionalidades do banco em uma única solução, tornando muito mais simples o acesso e manipulação do banco, podemos fazer absolutamente tudo e até mais do que faríamos usando Delphi VCL + IBX (ou qualquer outra suíte de acesso) e ao meu ver de forma mais fácil e transparente. Existem muitos detalhes do qual eu não tratei como isolamento de transações, configurações do servidor, também não falei sobre o namespace FirebirdSql.Data.Firebird.Isql que possui até um parser SQL entre outras classes bem úteis, é funcionalidade que não acaba mais, espero que tenham gostado deste artigo, abraços e até a próxima.
