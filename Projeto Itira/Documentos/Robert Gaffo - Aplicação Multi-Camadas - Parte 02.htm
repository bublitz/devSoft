Aplicação Multi-Camadas - Parte 02

Robert Gaffo (e-mail) é programador Delphi e instrutor de programação em Delphi e outras linguagens como Visual Basic, C/C++ e Java. Leciona aulas de conceitos de Banco de Dados, SQL Server, Oracle, Interbase/Firebird e MySql e dá treinamentos In Company. Trabalha com Delphi em aplicações desktop e multi-camadas.

Olá pessoal! Mais uma vez, vamos dar continuidade a esse artigo, visto que tenho recebido muitos e-mails a respeito dele. Nessa matéria discorreremos sobre como desenvolver o servidor de aplicação.

Observação: Nesse artigo foi usado o Delphi 2006.

Servidor de aplicação (Aplicativo servidor)

O servidor de aplicação nada mais que é que um programa que fica entre o aplicativo cliente e o sistema de gerenciamento de banco de dados (SGDB). Ele fica em uma máquina e todos os clientes se conectam a ele, o servidor de aplicação é quem se encarregará de acessar os dados do SGBD e passar em forma de pacote ao cliente que solicitou.

Observe que temos somente um aplicativo conectando ao SGBD, isso faz então com que consumamos somente uma licença de banco de dados e não uma licença para cada cliente, como é geralmente feito. Isso será abordado mais à frente.

Como funciona

Quando um aplicativo cliente é iniciado, ele é conectado ao servidor de aplicação (Remote DataModule), ativando o servidor de aplicação, caso não esteja ativado, e criando uma instância do Remote DataModule. Então, o aplicativo cliente faz uma solicitação de dados ao servidor de aplicação, que empacota esses dados e envia ao aplicativo cliente para que o usuário trabalhe com essa informação. O aplicativo cliente envia então essas modificações feitas pelo usuário em forma de pacote para que o servidor de aplicação possa fazer as atualizações dos dados.

O pacote enviado pelo aplicativo cliente chama-se Delta, que é uma propriedade do ClientDataSet que contém as alterações feitas pelo usuário, pode até ser usada para gerar logs de alterações no registro. O servidor de aplicação recebe esse pacote e tenta atualizar os dados nos registros originais, caso ninguém tenha feito alterações. Se alguém fez alguma alteração, o servidor de aplicação envia um erro que é capturado no evento OnReconcileError do ClientDataSet.

BDE e Multi-Camadas

Um dos problemas de se trabalhar com BDE, é que tem que ser instalado em todas as máquinas que o aplicativo cliente está. Quando trabalhamos com multi-camadas, isso não é necessário, pois, criamos um servidor de aplicação que vai se comunicar com o BDE e o BDE vai se comunicar com o banco de dados, a nossa aplicação cliente simplesmente vai se comunicar com o servidor de aplicação. Dessa forma, o BDE só será instalado na máquina que estiver o servidor de aplicação. Veja a estrutura abaixo:

Criando o servidor de aplicação

No nosso servidor de aplicação, teremos uma lista de todos os usuários que estão acessando o servidor, teremos também a interface com o SGBD.

Inicie uma nova aplicação no delphi. Desenhe a tela parecida com a abaixo:

Para facilitar a inserção e remoção dos usuários no ListBox, criaremos duas procedures para fazer isso.Declare a seguinte assinatura antes da sessão private:

Type

TForm1 = class(TForm)

...

procedure Inserir(Usuario : String);

procedure Remover(Usuario : String);

private

{ Private Declarations }

...

Pressione as teclas CTRL + SHIFT + C para gerar a implementação dessas procedures e digite o código abaixo:

procedure TForm1.Inserir(Usuario: String);

begin

lstUsuarios.Items.Add(Usuario)

end;

procedure TForm1.Remover(Usuario: String);

var

PosUsuario : Integer;

begin

PosUsuario := lstUsuarios.Items.IndexOf(Usuario);

if (PosUsuario > -1) then

lstUsuarios.Items.Delete(PosUsuario)

end;

Agora, vamos salvar nosso projeto. Vá em File à Save All ou pressione CTRL + SHIFT + S, salve a Unit1 como untMain e o projeto como ServerApplication.

Pronto. A tela do servidor já está pronta, agora vamos montar a estrutura de acesso a dados. Vá em File à New à Other, selecione Multitier e selecione Remote Data Module e clique em OK. Vai aparecer uma tela como a mostrada abaixo:

CoClass Name: É o nome e a classe do nosso Remote Data Module, esse é o nome do servidor que será usado para nossa aplicação cliente se conectar.

Instancing: Indica a forma de instanciar nosso servidor de aplicação.

. Internal: Indica que o modulo de dados é criado em um aplicativo servidor em processo, usado para criar um módulo de dados remoto como parte de uma DLL.

. Single Instance: Toda vez que um cliente se conectar em nosso servidor de aplicação, será aberto um executável para esse cliente, ou seja, uma instância de cada executável será criada dedicada a esse cliente. O módulo de dados remoto e o executável é único para cada cliente.

. Multiple Instance: Será um único executável, e toda vez que um cliente se conectar em nossa aplicação, será instanciado um módulo de dados remoto para ele, mas com o mesmo executável.

Threading Model: Especifica como os clientes que chamarão o servidor de aplicação e como será passada a requisição para a Interface do módulo de dados remoto.

. Single: O módulo de dados servirá uma requisição do cliente de cada vez.

. Apartment: Significa que cada instância do módulo de dados remoto servirá uma requisição de cada vez.

. Free: O módulo de dados remoto pode receber, simultaneamente, requisições de vários clientes, onde serão criados threads para atender essas requisições. Você deverá se assegurar que a instância dos dados e a memória global são protegidas contra conflitos de threads. Recomendável para quem usa tecnologia ADO.

. Both: Esse modelo é equivalente a Free, com a diferença que todas as chamadas para a interface dos aplicativos clientes serão serializadas.

Em CoClass name, digite Server e deixe as opções que já vem por default. Será criada uma tela parecida com a do DataModule que estamos acostumados a usar, a diferença é que o Remote DataModule permite conexão remota de outros aplicativos. Observe que o CoClass name que colocamos serve também como nome do módulo de dados remoto. Salve essa unit como untRDM, após você salvar essa unit, pedirá para você salvar o Type Library, mantenha o nome que é sugerido, que nada mais é do que o nome do nosso projeto.

Coloque também um DataModule. Para isso, vá em File à New à Other, selecione Delphi files e selecione DataModule e clique em OK. Em seguida, coloque um componente SQLConnection, dê ao DataModule o nome de DM e ao SQLConnection o nome de sqlConexao, vai ficar como mostrado abaixo:

Esse DataModule é necessário para que, quando ele criar uma nova instância do Remote DataModule, ele não crie também uma nova instância de conexão no servidor. Pois toda vez que um cliente se conecta em nosso servidor de aplicação, será criada uma instância do Remote DataModule na memória e, consecutivamente, uma instância de todos os objetos que estiverem nele. Se o SQLConnection, que é responsável pela conectar com a base de dados, estiver no Remote DataModule, será criada também uma nova instância dele e criando uma nova conexão com o banco. Por isso, separamos a conexão do banco em um DataModule.

Dê um duplo clique em sqlConexão. Crie uma nova conexão clicando no botão Add Connection . Em DriveName, escolha Interbase e em Connection Name coloque Servidor, ao clicar em OK vai ficar como mostrado abaixo:

Em Database, coloque o caminho onde foi salva a base de dados criada no artigo anterior. Clique no botão Test Connection para ver a situação da conexão. N ão se esqueça de iniciar o serviço do interbase, se não apareceu Successfully Connected, verifique os passos descritos acima novamente. Clique em OK na caixa de dialogo do dbExpress Connections voltando para o DataModule.

Certifique-se de que em Projects à Options na opção Forms o DataModule (DM) esteja na lista de autocreate forms e em primeiro lugar na lista. Salve essa unit com o nome untDM.

Agora que a conexão foi criada com sucesso, vamos colocar os componentes para acessar esses dados. Volte ao Remote DataModule e adicione um componente SQLDataSet (dbExpress) e dê o nome de sdsClientes e outro DataSetProvider (Data Access) com o nome de dspClientes. Pressione as teclas ALT + F11, que é a mesma coisa que File à Use Unit, escolha a unit do DataModule, que contém a conexão com a base de dados.

Selecione o componente sdsClientes e, na sua propriedades SQLConnection, selecione o componente sqlConexao que está no DataModule, selecione a propriedade CommandText e clique no botão ao lado, vai aparecer a seguinte tela:

Em Tables dê um duplo clique em clientes e em Fields dê um duplo clique no *. Observe que o comando SQL é montado ao lado, agora clique em OK. Coloque a propriedade Active para True para saber se está tudo ok, caso dê algum erro, verifique se não faltou nenhum dos passos descritos acima.

Vá ao menu View e selecione Type Library, através do Type Library podemos criar procedures e functions para a interface que o Delphi cria para o Remote DataModule, é através dessa interface que nossa aplicação cliente acessa nosso servidor de aplicação. Ao clicar em Type Library, vai aparecer uma tela como mostrada abaixo:

Selecione a opção IServer, ou seja, nossa interface, agora clique no botão New Method . Através do New Method, podemos criar uma procedure ou uma function, dê a esse método o nome de Login, selecione o método que apareceu e clique na guia parameters nas opções ao lado, vamos criar para esse método 2 parâmetros Usuario e Senha do tipo BSTR*. Quando a aplicação cliente se conectar ao servidor de aplicação, ela vai passar para o método Login da nossa interface o Usuario e a Senha para conexão.

Clique no botão Add, mais abaixo, para adicionar o outro parâmetro, o tipo BSTR * que nós digitamos é a mesma coisa que uma string do tipo WideString. É adotado esse nome genérico para que se tenha uma interface para que clientes de qualquer linguagem possam acessar também. Na coluna Modifier, indicamos se o parâmetro criado é de entrada, saída ou se ele retorna valores, esse último é usado para criarmos uma function e etc.

Crie um método para Logout do sistema somente com o parâmetro Usuario. Deve ficar como mostrado abaixo:

Pronto! Nós criamos os métodos com os respectivos parâmetros, então vamos implementar o método Login e o Logout. Salve sua aplicação e vá para Remote DataModule, é lá que fica a declaração dos métodos para implementação.

Procure na Unit a declaração do método, observe que o Delphi cria a assinatura da procedure. Se tivéssemos criado um parâmetro e em Modifier tivéssemos escolhido o tipo RetVal, retornar valores seria uma function. Observe que junto com a assinatura ele também gera a implementação, basta então codificarmos no espaço reservado.

Quando o cliente for fazer o Login na nossa aplicação, ele vai passar junto com a solicitação de conexão o Usuario e Senha. Com esses valores em mãos, você poderia, por exemplo, fazer uma consulta no banco e verificar se o usuário e senha digitados têm permissão para acessar o sistema ou outra validação que queira fazer. Nós só vamos adicionar o usuário logado no sistema no ListBox que está no formulário principal do nosso servidor, e limitar a quantidade de usuários que acessa nosso sistema, negando assim o acesso se estiver no limite. Quando o usuário sair do sistema, o mesmo emitirá um Logout para removermos o usuário da lista.

Digite o código abaixo para a implementação da rotina de Login:

if (Form1.lstUsuarios.Items.Count > 5) then

Raise Exception.Create('Limite de usuário excedido');

Form1.Inserir(Usuario);

Ou seja, estamos verificando se está no limite, senão estiver, ele adiciona o usuário no ListBox usando a procedure que nós criamos no formulário principal, caso esteja no limite, ele emite uma exceção (raise) bloqueando assim o acesso ao servidor.

Digite o código abaixo para a implementação da rotina de Logout.

Form1.Remover(Usuario);

Ou seja, estamos removendo o usuário do ListBox usando a procedure que nós criamos no formulário principal.

Pronto! Agora nosso servidor já está pronto, basta compilarmos e rodarmos o programa para que seja gerada um identificação (GUID) no registro Windows, é através dessa identificação que nosso cliente tem informações de nosso servidor.

Depois que você rodar o sistema, clique no menu Iniciar e escolha a opção Executar, digite RegEdit em executar para abrir o registro do Windows. Ao abrir o registro do Windows, clique no menu Editar e escolha a opção Localizar, procure por "Server". Conforme for pressionando a tecla F3 para localizar as ocorrências de Server, vão aparecendo várias opções que tem Server no meio, mas preste atenção que a chave localizada terá um valor e os dados desse valor devem estar Server Object, isso indica que é o servidor que nós criamos em Dephi. É através dessa chave que temos informações sobre o servidor.

Conclusão

Vimos em poucos e, relativamente, simples passos como podemos criar um servidor de aplicação e trabalhar com a tecnologia n-tier (multi-camadas). Dessa forma, conseguimos melhorar muito a qualidade, velocidade e controle de nossas aplicações, visto que isso também depende de como a rede foi implementada e de como o programador faz consultas ao banco de dados, note que no nosso exemplo usamos:

select * from clientes

O que não é nem um pouco recomendável, visto que gera tráfego na rede e pode tornar a aplicação e a rede lenta, o correto é sempre selecionarmos os dados com algum filtro para trazer o menor número de registros possível, pois nunca é necessário apresentar ou selecionar todos os registros do banco. Sempre especifique o nome dos campos que quer que apareça, mesmo que queira todos os campos da tabela. Como se trata de um exemplo, não precisamos entrar em maiores detalhes de performance.

Resumidamente, precisamos de um Remote DataModule e nele colocamos o SQLDataSet apontando para a conexão (SQLConnection) com a base de dados, que é aconselhável estar em um DataModule, e o DataSetProvider apontando para o SQLDataSet.

Na próxima parte, desenvolveremos a aplicação cliente que se comunicará com esse servidor.

Espero que esse artigo tenha ajudado a muitos. Até a próxima.
