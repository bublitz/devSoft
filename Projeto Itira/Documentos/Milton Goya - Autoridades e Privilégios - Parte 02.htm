Autoridades e Privilégios - Parte 02

Milton Goya (e-mail) foi desenvolver COBOL, CICS e DB2 de 1982 até 2001 em empresas como Brinquedos Estrela, Banco Itaú e Bradesco. Certificado OCP Oracle e DB2 e Professor/Instrutor na Faculdade de Tecnologia IBTA, e Centro de Treinamento IBTA. Faz também mestrado em ensino.

As autoridades e privilégios no DB2 são hierárquicas, é mais fácil perceber isso ao vermos a estrutura abaixo.

Como vimos na semana passada as autoridades são SYSADM, DBADM, SYSCTRL, SYSMAINT E LOAD. Nesta semana falaremos sobre os privilégios e na semana que vem abordaremos os privilégios dos desenvolvedores de aplicações.

Privilégios

Um privilégio é o direito que o usuário possui de criar ou acessar um objeto do banco de dados. São armazenados no catálogo do sistema e podem ser consultados em:

. SYSCAT.DBAUTH – Autoridades do banco;
. SYSCAT.TABAUTH – Privilégios de tabelas e views;
. SYSCAT.PACKAGEAUTH – Privilégios de pacotes;
. SYSCAT.INDEXAUTH – Privilégios de índices e
. SYSCAT.SCHEMAAUTH – Privilégios de esquema

São divididos em três tipos: proprietário, individual e implícito.

Privilégio de proprietário ou CONTROL : indica que o usuário tem acesso total a um objeto, ou seja, pode consulta-lo, altera-lo, elimina-lo e conceder privilégios sobre ele.

O privilégio de CONTROL é dado automaticamente para o criador do objeto e possuir o privilégio de CONTROL sobre um objeto é o mesmo que ser proprietário do mesmo. É possível conceder e revogar o privilégio de CONTROL através dos comandos GRANT e REVOKE.

GRANT CONTROL ON TABLE dept TO USER scott
GRANT CONTROL ON TABLE dept TO scott

Nos dois casos acima o usuário SCOTT está recebendo o privilégio de CONTROL na tabela dept. Um pouco mais à frente veremos o porque das duas sintaxes.

Privilégio individual: são os privilégios que permitem acesso a um objeto específico. Entre esses privilégios estão SELECT, DELETE, INSERT e UPDATE.

GRANT SELECT ON TABLE dept TO USER scott
GRANT SELECT ON TABLE dept TO scott

Nos dois casos acima o usuário SCOTT está recebendo privilégio para consultar a tabela dept.

Privilégio implícito: são os privilégios que são recebidos automaticamente quando são concedidos privilégios de alto nível para o usuário. Os privilégios implícitos NÃO são revogados quando o privilégio de alto nível é revogado do usuário. Por exemplo, quando um usuário recebe a autoridade DBADM ele recebe, implicitamente, os privilégios de CONNECT, CREATETAB, BINDADD, IMPLICIT_SCHEMA e CREATE_NOT_FENCED. Como já dito, caso a autoridade de DBADM seja revogada os privilégios implícitos continuarão atribuídos ao usuário.

O privilégio implícito também ocorre na execução de pacotes. Por exemplo, se um usuário não tem privilégio para fazer SELECT na tabela DEPT e o pacote executar uma consulta na DEPT ele passará a ter o privilégio necessário durante e apenas durante a execução do pacote. Dessa forma não é necessário que o usuário tenha acesso explícito a tabela DEPT. Esse tipo de privilégio é conhecido, também, como privilégio indireto .

Privilégios dos Objetos do Banco de Dados

Cada objeto do DB2 necessita de um privilégio ou de uma autoridade específica para serem utilizados. Existem as autoridades de banco de dados, privilégios de schemas , privilégios de tabelas, privilégios de views, privilégios de índice e privilégios de pacotes. Vamos examinar cada uma das situações:

Autoridades de Banco de Dados : Permitem criar e acessar um banco de dados. O criador do banco de dados recebe automaticamente a autoridade de DBADM. O DBADM pode executar qualquer comando SQL dentro do banco de dados. Quem possui a autoridade DBADM pode conceder privilégios de CONNECT, CREATETAB, IMPLICIT_SCHEMA, CREATE NOT FENCED.

O privilégio de IMPLICIT_SCHEMA permite que o usuário crie um objeto sem que seja necessário informar o nome do schema. O novo objeto irá assumir o schema do usuário que estiver criando o objeto. Durante a criação do banco de dados esse privilégio é concedido automaticamente ao grupo PUBLIC, ou seja, todos que se conectarem ao banco de dados podem criar um objeto sem que seja necessário informar o nome do schema.

Como devem ter percebido, quando um banco de dados é criado certos privilégios são concedidos automaticamente para o grupo PUBLIC. Caso o DBA queira revogar esses privilégios deve usar o comando REVOKE.

Privilégios de Schema : Neste privilégio estão contidas as permissões para criar, alterar ou eliminar objetos dentro de um esquema no banco de dados. O proprietário do esquema tem todas as permissões e pode repassa-las para outros usuários. Os principais objetos manipuláveis pelos esquemas são tabelas, views, índices, pacotes, UDTs, UDFs, triggers, stored procedures e aliás.

Privilégios de tabelas : Quem cria a tabela recebe implicitamente o privilégio de CONTROL na tabela, além disso, o SYSADM e o DBADM podem conceder o privilégio de CONTROL na tabela.

O privilégio de CONTROL permite adicionar novas colunas em uma tabela, criar chaves primárias, consultar, eliminar, alterar e acrescentar linhas, conceder esses privilégios a outros usuários, usar o utilitário de EXPORT, atualizar as estatísticas e outras atividades de manutenção como, por exemplo, reorganizar a tabela.

O privilégio de UPDATE on column permite que o usuário atualize apenas a coluna especificada. REFERENCE on column permite a criação de chave estrangeira em uma coluna específica.

Privilégios de view : Para criar uma view o usuário tem que ter, pelo menos, o privilégio de executar SELECT em cada uma das tabelas que compõe a view. Só é possível conceder os privilégios de DELETE, INSERT, SELECT e UPDATE na view se o usuário tiver o privilégio de CONTROL. A exceção a essa regra ocorre quando a view for read-only.

Privilégios de índice : Quem cria o índice recebe, automaticamente, o privilégio de CONTROL. O usuário só poderá criar o índice se tiver o privilégio de CREATE INDEX na tabela.

Privilégios de pacotes : Quem cria o pacote recebe o privilégio de CONTROL no pacote e pode conceder a outros usuários a capacidade de executar o pacote ou dar um rebind nele. No entanto, quem for executar o rebind precisa ter privilégios sobre os objetos referenciados nos comandos SQL, ou seja, se o pacote faz consulta a tabela dept quem for fazer o rebind tem que ter o privilégio de SELECT nessa tabela. Os privilégios devem ser concedidos ANTES de executar o rebind.

Para executar o rebind o usuário deve possuir o privilégio de BINDADD ou ser CONTROL do pacote ou pertencer ao grupo SYSADM ou DBADM.

Quem possui o privilégio de EXECUTE no pacote pode executa-lo sem ter o acesso explícito aos objetos que o pacote referencia, ou seja, se o pacote faz consulta a tabela dept o usuário NÃO precisa do privilégio explícito de SELECT na tabela dept.

Grant e Revoke

Existem dois comandos SQL que são usados para administra os privilégios:

. Grant: concede privilégios ao usuário.
. Revoke: revoga os privilégios do usuário.

Por exemplo, caso queiramos conceder o privilégio de SELECT na tabela dept para o usuário Scott, podemos usar o comando GRANT da seguinte forma:

GRANT SELECT ON TABLE dept TO Scott

A menos que o usuário Scott faça parte do grupo SYSADM ou DBADM ele não poderá repassar esse privilégio para outro usuário.

Para revogar o privilégio concedido usamos o comando REVOKE da seguinte forma:

REVOKE SELECT ON TABLE dept FROM Scott

Todos os privilégios de uma tabela podem ser removidos explicitamente através do comando REVOKE ALL, por exemplo:

REVOKE ALL ON dept FROM Scott

A mesma sintaxe pode ser usada para regovar todos os privilégios de um grupo.

Um usuário só pode conceder privilégios a um objeto do banco de dados nas seguintes circunstâncias:

. o objeto já existe;
. possui a autoridade SYSADM;
. possui a autoridade DBADM;
. possui o privilégio CONTROL;
. possui o privilégio WITH GRANT OPTION no objeto;
. somente SYSADM e DBADM podem conceder o privilégio CONTROL;
. somente o SYSADM pode conceder a autoridade DBADM.
